\chapter{Query Generation Approach}
\label{chap:query_generation}
\pagestyle{plain}

A learned cardinality estimator is only as good as the queries it trains on. Feed it a narrow, repetitive workload and it will memorize patterns instead of learning generalizable estimation. Feed it structurally diverse queries that cover a wide range of join topologies, predicate selectivities, and table combinations, and it stands a much better chance of producing useful estimates on unseen queries. This chapter describes how we generate such workloads, first through a large language model and then through a deterministic fallback, and how we ensure that every query reaching the training pipeline is valid, schema-compatible, and correctly formatted.


\section{Why Use a Language Model}

The traditional alternatives for obtaining training queries each have clear drawbacks. Query logs drawn from production systems carry privacy risks and tend to reflect the habits of a particular user base rather than the full space of possible queries. Template-based generators can produce any structure their templates define, but those templates must be written by hand and rarely capture the subtle correlations and predicate distributions found in real workloads. Uniform random sampling over the schema produces syntactically valid queries but distributes predicates and join patterns in ways that bear little resemblance to actual usage.

Large language models sidestep these issues. Models like Llama~3.2 have been trained on enormous corpora of code and natural language that include millions of SQL queries. When prompted with a schema and a set of generation constraints, they produce queries that look like something a human analyst might write: realistic predicate values, sensible join paths, and a natural mix of simple and complex patterns. The diversity comes for free, baked into the model's learned distribution over SQL, rather than requiring an engineer to enumerate templates.

We host the LLM locally through the Ollama platform. Local hosting means that no schema information or query data ever leaves the researcher's machine, the experiments do not depend on the availability or pricing of a third-party API, and the model version is fixed, which is important for reproducibility.


\section{Prompt Design}

The quality of what the LLM produces depends almost entirely on the prompt. Ours is structured as a five-part template, illustrated in Figure~\ref{fig:prompt_structure}. The first part assigns a role: ``You are a SQL query workload generator for a database.'' This framing primes the model to produce queries rather than explanations or tutorials. The second part injects the full database schema as DDL statements table names, columns, data types, foreign keys so the model knows what it is allowed to reference. The third part supplies column statistics: minimum and maximum values for numeric columns, which help the model pick predicate values that fall within realistic ranges rather than guessing wildly.

The fourth part is a list of hard constraints that keep the generated queries inside the fragment supported by the MSCN encoder. Every query must use \texttt{SELECT COUNT(*)}, involve one to three tables with proper aliases, join only on defined foreign key paths, and restrict filter predicates to simple comparisons (\texttt{=}, \texttt{<}, \texttt{>}) connected by \texttt{AND}. Constructs like \texttt{OR}, \texttt{IN}, \texttt{LIKE}, subqueries, and aggregations are forbidden. These rules are not arbitrary preferences; the MSCN encoding physically cannot represent queries that violate them, so letting such queries through would produce garbage features.

The fifth part specifies the output format: a JSON array where each element has a single \texttt{"sql"} key. Structured output like this makes parsing reliable. When the model does wrap extra text around the JSON, a regex-based fallback peels it away.


\section{Keeping the Workload Diverse}

Left to its own devices, an LLM tends to fall into patterns. After a few batches it may start recycling the same join structures or predicate styles, a problem sometimes called mode collapse. We counteract this with a rotating set of diversity hints. Five hints cycle in sequence: one pushes the model toward single-table queries with varied filters, another toward two-table joins, a third toward three-table joins, a fourth toward a mix of point lookups and range scans, and a fifth toward extreme selectivities queries returning either a handful of rows or millions. The hint index advances with each batch, so over the course of a full generation run every style gets equal representation.

Generating all queries in one prompt would blow past the model's context window and collapse diversity. Instead, the system works in batches. The total query target is divided by the batch size, and each batch gets a fresh prompt with the current diversity hint. Algorithm~\ref{alg:batch_generation} outlines the procedure.

\begin{algorithm}[htbp]
\caption{LLM-Based Query Generation}
\label{alg:batch_generation}
\begin{algorithmic}[1]
\State Determine how many batches are needed to reach the target number of queries
\For{each batch}
    \State Choose a diversity hint based on how many queries have been generated so far
    \State Construct a prompt containing the database schema, column statistics, and the chosen hint
    \For{up to 3 attempts}
        \State Send the prompt to the language model and receive a response
        \State Try to extract valid SQL queries from the response
        \If{at least one valid query was found}
            \State Add the extracted queries to the collection and move to the next batch
        \EndIf
    \EndFor
\EndFor
\State \Return the collected queries, trimmed to the target count
\end{algorithmic}
\end{algorithm}

Because LLM output is stochastic, any individual call can fail network hiccups, malformed JSON, the model hallucinating prose instead of SQL. Each batch is therefore retried up to three times. In practice this pushes the success rate close to one hundred percent.


\section{Filtering Out Bad Queries}

Even with careful prompting, some fraction of the LLM's output will be unusable. A query might reference a column that does not exist, use an \texttt{OR} that slipped past the instructions, or simply be unparseable. Rather than trusting the LLM to be perfect, we pass every query through four progressively stricter validation layers, illustrated in Figure~\ref{fig:validation_pipeline}.

The first layer is cheap: a set of regex checks that reject any query containing forbidden keywords like \texttt{OR}, \texttt{IN}, \texttt{LIKE}, or \texttt{BETWEEN}, as well as any query with more than one \texttt{SELECT} (a sign of subqueries). The second layer attempts to parse what survives, extracting the FROM clause, WHERE clause, and any JOIN\ldots ON clauses and decomposing them into tables, join conditions, and filter predicates.

The third layer cross-references the parsed result against the actual schema. Does every table name exist? Does every alias resolve to a real table? Does every predicate column belong to the table it claims to? Do the join conditions follow defined foreign key paths? Queries that fail any of these checks are discarded. The fourth layer converts the remaining queries into the structured format that MSCN expects: three comma-separated lists of tables, joins, and predicates, plus a placeholder for the cardinality that will be filled in during labeling.


\section{Synthetic Fallback}

Not every environment has access to an LLM, and not every experiment needs one. For quick iteration, unit testing, and baseline comparisons, the framework includes a deterministic synthetic generator. It works from hardcoded table definitions and uses weighted random sampling to decide how many tables each query involves, biasing toward two-table queries to match the star-schema structure of the benchmark database. The synthetic generator is fast, fully reproducible, and guaranteed to produce valid queries, but it cannot match the structural richness of LLM output.
