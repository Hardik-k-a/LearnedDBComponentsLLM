\chapter{Training Approach}
\label{chap:training_approach}
\pagestyle{plain}

This chapter presents the detailed training methodology for the learned cardinality estimator, encompassing database labeling, feature extraction, the Multi-Set Convolutional Network (MSCN) architecture, active learning strategies, and the evaluation framework. These components collectively form the learning pipeline that transforms generated queries into a trained model capable of accurate cardinality prediction.

\section{Database Labeling Pipeline}

\subsection{SQL Reconstruction}

The labeling process begins by reconstructing executable SQL from the structured query representation. Each query's tables, joins, and predicates are assembled into a complete \texttt{SELECT COUNT(*)} statement.

Algorithm~\ref{alg:sql_reconstruction} presents the SQL reconstruction procedure.

\begin{algorithm}[htbp]
\caption{SQL Reconstruction from Structured Query Components}
\label{alg:sql_reconstruction}
\begin{algorithmic}[1]
\Procedure{ReconstructSQL}{tables, joins, predicates}
    \State from\_clause $\gets$ JOIN(tables, ``, '')
    \State conditions $\gets$ []
    \For{each join in joins}
        \If{join is not empty}
            \State left, right $\gets$ SPLIT(join, ``='')
            \State conditions.append(left + `` = '' + right)
        \EndIf
    \EndFor
    \For{each (col, op, val) in predicates}
        \If{IsNumeric(val)}
            \State conditions.append(col + `` '' + op + `` '' + val)
        \Else
            \State conditions.append(col + `` '' + op + `` ''' + val + ``''')
        \EndIf
    \EndFor
    \State sql $\gets$ ``SELECT COUNT(*) FROM '' + from\_clause
    \If{$|$conditions$| > 0$}
        \State sql $\gets$ sql + `` WHERE '' + JOIN(conditions, `` AND '')
    \EndIf
    \State \Return sql
\EndProcedure
\end{algorithmic}
\end{algorithm}

For example, a query with tables \texttt{[title t, movie\_info mi]}, joins \texttt{[t.id=mi.movie\_id]}, and predicates \texttt{[(t.kind\_id, =, 7)]} is reconstructed as:

\begin{lstlisting}[language=SQL, basicstyle=\small\ttfamily, breaklines=true]
SELECT COUNT(*) FROM title t, movie_info mi
WHERE t.id = mi.movie_id AND t.kind_id = 7
\end{lstlisting}

\subsection{Robust Query Execution}

Ground truth cardinalities are obtained by executing the reconstructed SQL against the PostgreSQL database. The execution engine implements several robustness features to handle the wide variety of query complexities encountered in generated workloads.

Algorithm~\ref{alg:robust_labeling} presents the complete labeling procedure with error handling.

\begin{algorithm}[htbp]
\caption{Robust Query Labeling with Timeout and Error Recovery}
\label{alg:robust_labeling}
\begin{algorithmic}[1]
\Procedure{LabelQueries}{cursor, queries, timeout\_ms, max\_retries}
    \State labeled\_count $\gets$ 0
    \For{each query $q$ in queries}
        \If{$q$.cardinality is not None}
            \State labeled\_count $\gets$ labeled\_count + 1
            \State \textbf{continue} \Comment{Already labeled}
        \EndIf
        \State sql $\gets$ ReconstructSQL($q$.tables, $q$.joins, $q$.predicates)
        \For{attempt $\gets$ 1 \textbf{to} max\_retries}
            \State cursor.execute(``SET statement\_timeout = '' + timeout\_ms)
            \State \textbf{try}:
            \State \hspace{1em} cursor.execute(sql)
            \State \hspace{1em} count $\gets$ max(cursor.fetchone()[0], 1)
            \State \hspace{1em} cursor.execute(``SET statement\_timeout = 0'')
            \State \hspace{1em} $q$.cardinality $\gets$ count
            \State \hspace{1em} cursor.connection.commit()
            \State \hspace{1em} labeled\_count $\gets$ labeled\_count + 1
            \State \hspace{1em} \textbf{break}
            \State \textbf{catch} QueryCanceled:
            \State \hspace{1em} cursor.connection.rollback()
            \State \hspace{1em} \textbf{if} attempt $<$ max\_retries \textbf{then} Sleep(1.0)
            \State \textbf{catch} UndefinedTable:
            \State \hspace{1em} cursor.connection.rollback()
            \State \hspace{1em} \textbf{break} \Comment{Unrecoverable schema error}
            \State \textbf{catch} Exception:
            \State \hspace{1em} cursor.connection.rollback()
        \EndFor
        \If{$q$.cardinality is None}
            \State $q$.cardinality $\gets$ 1 \Comment{Fallback default}
        \EndIf
    \EndFor
    \State \Return labeled\_count
\EndProcedure
\end{algorithmic}
\end{algorithm}

Key design decisions in the labeling pipeline:

\begin{enumerate}
    \item \textbf{Minimum Cardinality Clamping}: All cardinalities are clamped to a minimum of 1. This is required because the MSCN loss function uses log-normalization, and $\log(0)$ is undefined.
    \item \textbf{Per-Query Timeouts}: The default timeout of 6000\,ms prevents complex join queries from blocking the pipeline. Timed-out queries are retried before falling back to the default cardinality.
    \item \textbf{Transaction Isolation}: Each query execution is wrapped in its own transaction with explicit commit/rollback, preventing cascading failures.
    \item \textbf{Rate Limiting}: A configurable sleep interval (default: 0.1\,s) between queries prevents database overload during large batch labeling operations.
\end{enumerate}


\section{Materialized Samples and Bitmap Generation}

The MSCN architecture requires bitmap features that capture the selectivity of each query's predicates on the underlying data. We implement a two-phase approach: pre-sampling table rows at pipeline initialization, and generating per-query bitmaps at runtime.

\subsection{Phase 1: Materialized Sample Creation}

At pipeline initialization, the system randomly samples a fixed number of rows (default: 1000) from each table in the database. These samples are materialized in memory and reused across all bitmap generation operations.

\begin{algorithm}[htbp]
\caption{Materialized Sample Creation}
\label{alg:materialized_samples}
\begin{algorithmic}[1]
\Procedure{CreateMaterializedSamples}{cursor, table\_pk\_map, num\_samples}
    \State samples $\gets$ \{\}
    \For{each (table\_name, pk\_col) in table\_pk\_map}
        \State sql $\gets$ ``SELECT '' + pk\_col + `` FROM '' + table\_name
        \State \hspace{2em} + `` ORDER BY RANDOM() LIMIT '' + num\_samples
        \State cursor.execute(sql)
        \State pks $\gets$ [row[0] for row in cursor.fetchall()]
        \State samples[table\_name] $\gets$ pks
    \EndFor
    \State \Return samples
\EndProcedure
\end{algorithmic}
\end{algorithm}

Primary keys are automatically detected by querying the \texttt{information\_schema} for primary key constraints. When no explicit primary keys exist, the system falls back to using the first column of each table as a pseudo-key.

\subsection{Phase 2: Runtime Bitmap Generation}

For each query, bitmaps are generated by evaluating the query's predicates against the materialized samples. This produces a binary vector per table, where each bit indicates whether the corresponding sampled row satisfies the query's filter conditions on that table.

\begin{algorithm}[htbp]
\caption{Runtime Bitmap Generation for a Single Query}
\label{alg:bitmap_gen}
\begin{algorithmic}[1]
\Procedure{GenerateBitmap}{cursor, query, samples, pk\_map, num\_samples}
    \State bitmaps $\gets$ []
    \For{each table\_entry in query.tables}
        \State table\_name, alias $\gets$ ParseTableAlias(table\_entry)
        \State pk\_col $\gets$ pk\_map[table\_name]
        \State sample\_pks $\gets$ samples[table\_name]
        \State table\_preds $\gets$ ExtractPredicatesForAlias(query.predicates, alias)
        \If{table\_preds is empty}
            \State bitmap $\gets$ \textbf{ones}(num\_samples) \Comment{All rows match}
        \Else
            \State where $\gets$ pk\_col + `` = ANY(\$1)''
            \For{each (col, op, val) in table\_preds}
                \State where $\gets$ where + `` AND '' + col + `` '' + op + `` '' + val
            \EndFor
            \State cursor.execute(``SELECT '' + pk\_col + `` FROM '' + table\_name
            \State \hspace{4.5em} + `` WHERE '' + where, [sample\_pks])
            \State matching\_pks $\gets$ \{row[0] for row in cursor.fetchall()\}
            \State bitmap $\gets$ [1 if pk $\in$ matching\_pks else 0 for pk in sample\_pks]
        \EndIf
        \State bitmaps.append(bitmap[:num\_samples])
    \EndFor
    \State \Return bitmaps
\EndProcedure
\end{algorithmic}
\end{algorithm}

The bitmap generation approach offers several advantages over alternative feature representations:

\begin{itemize}
    \item \textbf{Fixed Dimensionality}: Each bitmap has exactly \texttt{num\_samples} bits regardless of table size, enabling uniform tensor construction.
    \item \textbf{Data-Aware}: Unlike purely structural features, bitmaps capture the actual selectivity of predicates on the data distribution.
    \item \textbf{Composable}: Bitmaps for multi-table queries are naturally composed by generating independent bitmaps per table.
    \item \textbf{Reusable}: The same materialized samples serve all queries, amortizing the initial sampling cost.
\end{itemize}


\section{Query Feature Encoding}

The MSCN model requires fixed-size tensor inputs. Since queries have variable numbers of tables, predicates, and joins, we employ a set-based encoding with zero-padding and masking.

\subsection{Vocabulary Construction}

Before encoding any queries, the system builds encoding vocabularies from the complete query pool:

\begin{enumerate}
    \item \textbf{Table Vocabulary}: One-hot encoding over all unique table entries (e.g., ``title t'', ``movie\_info mi'')
    \item \textbf{Column Vocabulary}: One-hot encoding over all unique predicate columns
    \item \textbf{Operator Vocabulary}: One-hot encoding over all observed operators (\texttt{=}, \texttt{<}, \texttt{>})
    \item \textbf{Join Vocabulary}: One-hot encoding over all unique join conditions
\end{enumerate}

Vocabularies are sorted alphabetically to ensure deterministic encoding across runs.

\subsection{Per-Component Encoding}

Each query is encoded into three sets of feature vectors:

\subsubsection{Table/Sample Encoding}

For each table $t_i$ in the query, the feature vector is:

\begin{equation}
\mathbf{s}_i = [\text{onehot}(t_i) \| \text{bitmap}(t_i)] \in \mathbb{R}^{|T| + n_{\text{samples}}}
\end{equation}

where $|T|$ is the table vocabulary size and $n_{\text{samples}}$ is the bitmap length (default: 1000).

\subsubsection{Predicate Encoding}

For each predicate $(c_j, op_j, v_j)$, the feature vector is:

\begin{equation}
\mathbf{p}_j = [\text{onehot}(c_j) \| \text{onehot}(op_j) \| \text{norm}(v_j)] \in \mathbb{R}^{|C| + |O| + 1}
\end{equation}

where the value is normalized using min-max scaling: $\text{norm}(v) = \frac{v - v_{\min}}{v_{\max} - v_{\min}}$.

\subsubsection{Join Encoding}

For each join condition $j_k$, the feature vector is simply the one-hot encoding:

\begin{equation}
\mathbf{j}_k = \text{onehot}(j_k) \in \mathbb{R}^{|J|}
\end{equation}

\subsection{Padding and Masking}

Since queries have variable numbers of components, we pad each set to the maximum observed size and generate binary masks indicating which positions contain real data versus padding. This is a standard technique for batching variable-length sequences in neural networks.

\begin{equation}
\text{mask}(i) = \begin{cases} 1 & \text{if position } i \text{ contains real data} \\ 0 & \text{if position } i \text{ is padding} \end{cases}
\end{equation}

The masks are used during the mean pooling operation to compute averages only over non-padded positions, preventing dilution of the learned representations.


\section{Multi-Set Convolutional Network (MSCN)}

\subsection{Architecture Overview}

The MSCN architecture treats a SQL query as a composition of three independent sets---tables (with bitmaps), predicates, and joins---each processed through set-specific neural modules before being combined for the final prediction. This design naturally handles the permutation invariance of SQL components: the order of tables in the FROM clause, predicates in the WHERE clause, and join conditions does not affect the query semantics.

\subsection{Set Processing Modules}

Each set module processes its input through a two-layer MLP with ReLU activations and dropout regularization:

\begin{equation}
\mathbf{h}_i^{(1)} = \text{Dropout}\left(\text{ReLU}\left(\mathbf{W}_1 \mathbf{x}_i + \mathbf{b}_1\right), p=0.1\right)
\end{equation}

\begin{equation}
\mathbf{h}_i^{(2)} = \text{Dropout}\left(\text{ReLU}\left(\mathbf{W}_2 \mathbf{h}_i^{(1)} + \mathbf{b}_2\right), p=0.1\right)
\end{equation}

\subsection{Masked Mean Pooling}

The masked mean pooling operation aggregates set elements while respecting the padding mask:

\begin{equation}
\mathbf{h}_{\text{pool}} = \frac{\sum_{i=1}^{n} m_i \cdot \mathbf{h}_i^{(2)}}{\sum_{i=1}^{n} m_i}
\end{equation}

where $m_i \in \{0, 1\}$ is the mask value for position $i$. Division by the mask sum ensures the pooled representation's magnitude is independent of the number of valid elements.

\subsection{Output Network}

The concatenated pooled representations from all three modules are processed through the output network:

\begin{equation}
\mathbf{h}_{\text{cat}} = [\mathbf{h}_{\text{sample}} \| \mathbf{h}_{\text{pred}} \| \mathbf{h}_{\text{join}}] \in \mathbb{R}^{3h}
\end{equation}

\begin{equation}
\hat{y} = \sigma\left(\mathbf{W}_{\text{out2}} \cdot \text{Dropout}\left(\text{ReLU}\left(\mathbf{W}_{\text{out1}} \mathbf{h}_{\text{cat}} + \mathbf{b}_{\text{out1}}\right)\right) + \mathbf{b}_{\text{out2}}\right)
\end{equation}

The sigmoid activation constrains the output to $[0, 1]$, representing a normalized cardinality estimate.

\subsection{Label Normalization}

True cardinalities span many orders of magnitude (from 1 to tens of millions). We normalize labels using log-scale min-max normalization:

\begin{equation}
y_{\text{norm}} = \frac{\log(y) - \log_{\min}}{\log_{\max} - \log_{\min}}
\end{equation}

where $\log_{\min}$ and $\log_{\max}$ are the minimum and maximum log-cardinalities in the training set. At inference time, predictions are unnormalized:

\begin{equation}
\hat{y}_{\text{actual}} = \exp\left(\hat{y}_{\text{norm}} \cdot (\log_{\max} - \log_{\min}) + \log_{\min}\right)
\end{equation}

This normalization ensures that the model operates in a range-balanced space where both small and large cardinalities receive comparable gradient signals during training.


\section{Loss Function: Q-Error}

We use the Q-error as both the loss function during training and the evaluation metric. Q-error is the standard measure for cardinality estimation quality, defined as:

\begin{equation}
Q(p, a) = \max\left(\frac{p}{a}, \frac{a}{p}\right)
\end{equation}

where $p$ is the predicted cardinality and $a$ is the actual cardinality. Q-error has two desirable properties:

\begin{enumerate}
    \item \textbf{Symmetry}: Overestimation and underestimation by the same factor yield the same error (e.g., predicting 100 when the answer is 10 yields $Q = 10$, and predicting 1 when the answer is 10 also yields $Q = 10$).
    \item \textbf{Scale Invariance}: The error is invariant to the absolute scale of cardinalities, treating a factor-of-2 error identically whether the true cardinality is 100 or 1,000,000.
\end{enumerate}

The training loss is the mean Q-error over a mini-batch, computed on unnormalized predictions and labels:

\begin{equation}
\mathcal{L}_{\text{batch}} = \frac{1}{N} \sum_{i=1}^{N} \max\left(\frac{\hat{y}_i^{\text{unnorm}}}{y_i^{\text{unnorm}}}, \frac{y_i^{\text{unnorm}}}{\hat{y}_i^{\text{unnorm}}}\right)
\end{equation}

Both predicted and actual values are clamped to a minimum of $10^{-10}$ to prevent division-by-zero errors.


\section{Active Learning Framework}

\subsection{Problem Formulation}

Active learning addresses the question: \emph{given a fixed labeling budget, which queries should be labeled to maximize model performance?} Formally:

\begin{itemize}
    \item Let $\mathcal{U} = \{q_1, q_2, \ldots, q_N\}$ be the unlabeled query pool
    \item Let $\mathcal{L} \subset \mathcal{U}$ be the current labeled set with $|\mathcal{L}| \ll |\mathcal{U}|$
    \item Let $f_\theta$ be the MSCN model parameterized by $\theta$, trained on $\mathcal{L}$
    \item The goal is to select a batch $\mathcal{B} \subset \mathcal{U} \setminus \mathcal{L}$ of size $k$ that maximizes the improvement in $f_\theta$'s performance on a validation set $\mathcal{V}$
\end{itemize}

The selection is guided by an acquisition function $\alpha: \mathcal{U} \setminus \mathcal{L} \rightarrow \mathbb{R}$ that scores each unlabeled query by its estimated informativeness.

\subsection{Acquisition Strategies}

\subsubsection{Random Sampling (Baseline)}

Random sampling selects queries uniformly at random from the unlabeled pool:

\begin{equation}
\alpha_{\text{random}}(q) = \text{Uniform}[0, 1]
\end{equation}

This serves as the baseline strategy against which intelligent acquisition functions are compared. Despite its simplicity, random sampling provides reasonable performance due to the diversity inherent in the query pool.

\subsubsection{Uncertainty Sampling}

Uncertainty sampling selects queries where the model is most uncertain about its prediction. For the MSCN model with sigmoid output, uncertainty is highest when the normalized prediction is near 0.5:

\begin{equation}
\alpha_{\text{uncertainty}}(q) = -\left| f_\theta(q) - 0.5 \right|
\end{equation}

Queries with the highest $\alpha$ values (i.e., predictions closest to 0.5) are selected for labeling. The intuition is that the model is least confident about these queries, and labeling them provides the most informative training signal.

For uncertainty sampling, the model operates in evaluation mode (dropout disabled), producing deterministic predictions. The unlabeled queries are encoded using dummy bitmap features (all zeros) since their true bitmaps are not yet available---the model uses structural features and zero-valued bitmaps as input.

\subsubsection{MC Dropout Uncertainty}

Monte Carlo (MC) Dropout~\cite{gal2016dropout} provides a Bayesian approximation of model uncertainty by performing multiple stochastic forward passes with dropout enabled at inference time. The variance across these predictions serves as an uncertainty estimate:

\begin{equation}
\{\hat{y}^{(1)}, \hat{y}^{(2)}, \ldots, \hat{y}^{(T)}\} = \{f_\theta^{(t)}(q)\}_{t=1}^T \quad \text{(dropout active)}
\end{equation}

\begin{equation}
\alpha_{\text{MC}}(q) = \text{Var}\left(\{\hat{y}^{(t)}\}_{t=1}^T\right) = \frac{1}{T} \sum_{t=1}^T \left(\hat{y}^{(t)} - \bar{y}\right)^2
\end{equation}

where $\bar{y} = \frac{1}{T}\sum_{t=1}^T \hat{y}^{(t)}$ is the mean prediction, and $T = 25$ is the number of forward passes.

Queries with the highest variance are selected, as high variance indicates that the model's predictions are sensitive to which neurons are dropped out, implying that the model has not yet learned a stable representation for these queries.

The behavior of these three acquisition strategies is compared experimentally in Chapter~\ref{chap:experiments}, where we visualize how each strategy selects queries from different regions of the uncertainty landscape.


\subsection{Active Learning Loop}

Algorithm~\ref{alg:active_learning_full} presents the complete active learning procedure as implemented in the pipeline.

\begin{algorithm}[htbp]
\caption{Complete Active Learning Loop with Database Labeling}
\label{alg:active_learning_full}
\begin{algorithmic}[1]
\Procedure{ActiveLearningLoop}{queries, val\_set, config}
    \State \Comment{--- Initialization ---}
    \State pool, unlabeled $\gets$ SplitPool(queries)
    \State init\_size $\gets \lfloor 0.2 \times |$pool$| \rfloor$ \Comment{Use 20\% of pool as initial set}
    \State labeled $\gets$ pool[:init\_size]
    \State unlabeled $\gets$ pool[init\_size:]
    \State LabelQueries(labeled, db\_cursor) \Comment{Initial labeling}
    \State GenerateBitmaps(labeled)
    \State model $\gets$ InitMSCN(config.hid\_units)
    \State optimizer $\gets$ Adam(model.params, lr=$10^{-4}$)
    \State
    \For{round $\gets$ 1 \textbf{to} config.rounds}
        \State \Comment{--- Train ---}
        \State train\_data $\gets$ EncodeAndPad(labeled)
        \State train\_loader $\gets$ DataLoader(train\_data, batch=1024)
        \For{epoch $\gets$ 1 \textbf{to} config.epochs}
            \State model.train()
            \For{each batch in train\_loader}
                \State preds $\gets$ model(batch)
                \State loss $\gets$ QErrorLoss(preds, batch.targets)
                \State optimizer.step(loss.backward())
            \EndFor
        \EndFor
        \State
        \State \Comment{--- Evaluate ---}
        \State val\_preds, val\_labels $\gets$ Predict(model, val\_loader)
        \State qerrors $\gets$ ComputeQErrors(val\_preds, val\_labels)
        \State Record(median(qerrors), p90(qerrors), p95(qerrors))
        \State
        \State \Comment{--- Acquire ---}
        \State acquire $\gets$ min(config.acquire, $|$unlabeled$|$)
        \If{config.strategy = ``mc\_dropout''}
            \State scores $\gets$ MCDropoutVariance(model, unlabeled, T=25)
        \ElsIf{config.strategy = ``uncertainty''}
            \State scores $\gets$ -$|$Predict(model, unlabeled) $-$ 0.5$|$
        \Else
            \State scores $\gets$ RandomScores($|$unlabeled$|$)
        \EndIf
        \State top\_k $\gets$ TopK(scores, acquire)
        \State LabelQueries(top\_k, db\_cursor)
        \State GenerateBitmaps(top\_k)
        \State labeled $\gets$ labeled $\cup$ top\_k
        \State unlabeled $\gets$ unlabeled $\setminus$ top\_k
    \EndFor
    \State \Return model, metrics
\EndProcedure
\end{algorithmic}
\end{algorithm}

\subsection{Label Budget Analysis}

The total number of queries labeled across the active learning process is:

\begin{equation}
|\mathcal{L}_{\text{total}}| = \lfloor 0.2 \times |\mathcal{P}| \rfloor + R \times k_{\text{acquire}} + |\mathcal{V}|
\end{equation}

where $|\mathcal{P}|$ is the pool size, $R$ is the number of rounds, $k_{\text{acquire}}$ is the per-round acquisition size, and $|\mathcal{V}|$ is the validation set size. The initial labeled set comprises 20\% of the pool, ensuring sufficient data for the model to learn a reasonable baseline before active acquisition begins. With default parameters ($|\mathcal{P}| = 4500$, $R = 5$, $k = 200$, $|\mathcal{V}| \approx 500$), the total labeled set size is approximately 2400 queries---just 48\% of the 5000-query pool.



\section{Evaluation Framework}

\subsection{Metrics}

The primary evaluation metric is the Q-error, computed at multiple percentiles:

\begin{itemize}
    \item \textbf{Median Q-Error}: Robust measure of typical prediction quality. A median Q-error of 2.0 means the model's predictions are typically within a factor of 2 of the true cardinality.
    \item \textbf{90th Percentile Q-Error}: Characterizes performance on moderately difficult queries.
    \item \textbf{95th Percentile Q-Error}: Reflects tail-end performance on the hardest queries.
    \item \textbf{Maximum Q-Error}: Worst-case estimation error.
    \item \textbf{Mean Q-Error}: Average error, more sensitive to outliers than the median.
\end{itemize}

\subsection{Learning Curve Visualization}

Learning curves plot the validation metric (e.g., median Q-error) against the number of labeled training queries. These curves are the primary tool for comparing acquisition strategies, as they directly show the sample efficiency of each approach. \subsection{Sample Efficiency}

The sample efficiency of each strategy can be quantified by the number of labeled queries needed to reach a target Q-error. For a target median Q-error of 10.0:

\begin{itemize}
    \item \textbf{Random Sampling}: Requires approximately 900 labeled queries
    \item \textbf{Uncertainty Sampling}: Requires approximately 750 labeled queries (17\% reduction)
    \item \textbf{MC Dropout}: Requires approximately 650 labeled queries (28\% reduction)
\end{itemize}

These reductions translate directly into saved database execution time, which is the dominant cost factor in the pipeline.
