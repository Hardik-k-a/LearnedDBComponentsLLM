\chapter{Training Approach}
\label{chap:training_approach}
\pagestyle{plain}

This chapter presents the training methodology for the learned cardinality estimator, encompassing database labeling, feature extraction, the Multi-Set Convolutional Network architecture, active learning strategies, and the evaluation framework. Together, these components transform generated queries into a trained model capable of accurate cardinality prediction.


\section{Database Labeling}

Ground truth cardinalities are obtained by executing reconstructed \texttt{SELECT COUNT(*)} queries against the target database. Each query's tables, joins, and predicates are assembled into a complete SQL statement and submitted to the database engine. Since generated workloads contain queries of widely varying complexity, some involving expensive multi-table joins over tens of millions of rows, the labeling pipeline enforces per-query timeouts and implements retry logic to ensure that no single query can block the pipeline. Queries that fail after all retries are assigned a fallback cardinality of one, satisfying the log-normalization requirement of the model while allowing the pipeline to continue. All cardinalities are clamped to a minimum of one because the training loss operates in log-space, where $\log(0)$ is undefined.

The labeling stage is the most computationally expensive part of the pipeline. A single query on the IMDB dataset can take several seconds to execute, and labeling thousands of queries may require hours of database time. This cost is the primary motivation for active learning: by intelligently selecting which queries to label, the framework achieves strong model performance while executing only a fraction of the total query pool.


\section{Feature Extraction and Bitmap Generation}

The MSCN architecture requires fixed-size tensor inputs, but queries have variable numbers of tables, predicates, and joins. The feature extraction stage addresses this through set-based encoding with zero-padding and masking.

Before encoding any queries, the system builds encoding vocabularies from the complete query pool. A table vocabulary provides one-hot encodings over all unique table entries, a column vocabulary covers all unique predicate columns, an operator vocabulary covers the observed comparison operators, and a join vocabulary covers all unique join conditions. These vocabularies are sorted alphabetically to ensure deterministic encoding across runs.

Each query is then encoded into three sets of feature vectors. For each table $t_i$ in the query, the feature vector concatenates the table's one-hot encoding with a bitmap vector that captures the selectivity of the query's predicates on a pre-sampled set of rows:

\begin{equation}
\mathbf{s}_i = [\text{onehot}(t_i) \| \text{bitmap}(t_i)] \in \mathbb{R}^{|T| + n_{\text{samples}}}
\end{equation}

For each predicate $(c_j, op_j, v_j)$, the feature vector concatenates a column one-hot vector, an operator one-hot vector, and the normalized predicate value:

\begin{equation}
\mathbf{p}_j = [\text{onehot}(c_j) \| \text{onehot}(op_j) \| \text{norm}(v_j)] \in \mathbb{R}^{|C| + |O| + 1}
\end{equation}

where the value is normalized using min-max scaling: $\text{norm}(v) = \frac{v - v_{\min}}{v_{\max} - v_{\min}}$. Each join condition $j_k$ is simply represented as its one-hot encoding $\mathbf{j}_k = \text{onehot}(j_k) \in \mathbb{R}^{|J|}$.

Since queries have variable numbers of components, each set is padded to the maximum observed size and binary masks are generated to indicate which positions contain real data versus padding:

\begin{equation}
\text{mask}(i) = \begin{cases} 1 & \text{if position } i \text{ contains real data} \\ 0 & \text{if position } i \text{ is padding} \end{cases}
\end{equation}

The masks are used during the mean pooling operation to compute averages only over non-padded positions, preventing dilution of the learned representations.

The bitmap features deserve special attention, as they are the mechanism through which the model gains awareness of the underlying data distribution. At pipeline initialization, a fixed set of rows is randomly sampled from each table and materialized in memory. For each query, the system evaluates which of these sampled rows satisfy the query's filter predicates on each table, producing a binary vector where each bit indicates whether the corresponding sampled row passes the filter. This approach offers fixed dimensionality regardless of table size, composability across multi-table queries, and reusability of the same materialized samples for all queries in the workload.


\section{Multi-Set Convolutional Network}

The MSCN architecture, proposed by Kipf et al.~\cite{DBLP:conf/cidr/KipfKRLBK19}, treats a SQL query as a composition of three independent sets---tables with bitmaps, predicates, and joins---each processed through set-specific neural modules before being combined for the final prediction. This design naturally handles the permutation invariance of SQL components: the order of tables in the FROM clause, predicates in the WHERE clause, and join conditions does not affect the query semantics. Each set module processes its input through a two-layer MLP with ReLU activations and dropout regularization. The first layer projects the input features to a hidden dimension $h$, and the second layer refines the representation:

\begin{equation}
\mathbf{h}_i^{(1)} = \text{Dropout}\left(\text{ReLU}\left(\mathbf{W}_1 \mathbf{x}_i + \mathbf{b}_1\right), p\right)
\end{equation}

\begin{equation}
\mathbf{h}_i^{(2)} = \text{Dropout}\left(\text{ReLU}\left(\mathbf{W}_2 \mathbf{h}_i^{(1)} + \mathbf{b}_2\right), p\right)
\end{equation}

The masked mean pooling operation then aggregates the set elements while respecting the padding mask, computing the average only over positions containing real data:

\begin{equation}
\mathbf{h}_{\text{pool}} = \frac{\sum_{i=1}^{n} m_i \cdot \mathbf{h}_i^{(2)}}{\sum_{i=1}^{n} m_i}
\end{equation}

The concatenated pooled representations from all three modules pass through an output network that produces the final normalized cardinality estimate:

\begin{equation}
\hat{y} = \sigma\left(\mathbf{W}_{\text{out2}} \cdot \text{Dropout}\left(\text{ReLU}\left(\mathbf{W}_{\text{out1}} [\mathbf{h}_{\text{sample}} \| \mathbf{h}_{\text{pred}} \| \mathbf{h}_{\text{join}}] + \mathbf{b}_{\text{out1}}\right)\right) + \mathbf{b}_{\text{out2}}\right)
\end{equation}

The sigmoid activation constrains the output to $[0, 1]$, representing a normalized cardinality estimate.


\section{Label Normalization and Loss Function}

True cardinalities span many orders of magnitude, from single-digit counts to tens of millions. Passing raw integer values to the loss function would cause catastrophic gradient imbalances, as the model would be dominated by large-cardinality queries. We normalize labels using log-scale min-max normalization:

\begin{equation}
y_{\text{norm}} = \frac{\log(y) - \log_{\min}}{\log_{\max} - \log_{\min}}
\end{equation}

where $\log_{\min}$ and $\log_{\max}$ are the minimum and maximum log-cardinalities in the training set. At inference time, predictions are unnormalized by inverting this transformation:

\begin{equation}
\hat{y}_{\text{actual}} = \exp\left(\hat{y}_{\text{norm}} \cdot (\log_{\max} - \log_{\min}) + \log_{\min}\right)
\end{equation}

We use Q-error as both the training loss and the evaluation metric. Q-error is the standard measure for cardinality estimation quality, defined as:

\begin{equation}
Q(p, a) = \max\left(\frac{p}{a}, \frac{a}{p}\right)
\end{equation}

where $p$ is the predicted cardinality and $a$ is the actual cardinality. Q-error possesses two desirable properties: it is symmetric, meaning that overestimation and underestimation by the same factor yield the same error, and it is scale-invariant, treating a factor-of-two error identically whether the true cardinality is one hundred or one million. The training loss is the mean Q-error over a mini-batch, computed on unnormalized predictions and labels.


\section{Active Learning Framework}

Active learning addresses the central question of this work: given a fixed labeling budget, which queries should be labeled to maximize model performance? Formally, let $\mathcal{U} = \{q_1, q_2, \ldots, q_N\}$ be the unlabeled query pool, let $\mathcal{L} \subset \mathcal{U}$ be the current labeled set with $|\mathcal{L}| \ll |\mathcal{U}|$, and let $f_\theta$ be the MSCN model trained on $\mathcal{L}$. The goal is to select a batch $\mathcal{B} \subset \mathcal{U} \setminus \mathcal{L}$ of size $k$ that maximizes the improvement in $f_\theta$'s performance on a held-out validation set $\mathcal{V}$. The selection is guided by an acquisition function $\alpha: \mathcal{U} \setminus \mathcal{L} \rightarrow \mathbb{R}$ that scores each unlabeled query by its estimated informativeness.

The framework supports three acquisition strategies. Random sampling serves as the baseline, selecting queries uniformly at random from the unlabeled pool with $\alpha_{\text{random}}(q) = \text{Uniform}[0, 1]$. Despite its simplicity, random sampling provides reasonable performance due to the inherent diversity of the query pool.

Uncertainty sampling selects queries where the model is most uncertain about its prediction. For the MSCN model with sigmoid output, uncertainty is highest when the normalized prediction is near 0.5, yielding $\alpha_{\text{uncertainty}}(q) = -| f_\theta(q) - 0.5 |$. The intuition is that the model is least confident about these queries, and labeling them provides the most informative training signal.

Monte Carlo Dropout~\cite{gal2016dropout} provides a Bayesian approximation of model uncertainty by performing $T$ stochastic forward passes with dropout enabled at inference time. The variance across these predictions serves as the uncertainty estimate:

\begin{equation}
\alpha_{\text{MC}}(q) = \text{Var}\left(\{\hat{y}^{(t)}\}_{t=1}^T\right) = \frac{1}{T} \sum_{t=1}^T \left(\hat{y}^{(t)} - \bar{y}\right)^2
\end{equation}

where $\bar{y} = \frac{1}{T}\sum_{t=1}^T \hat{y}^{(t)}$ is the mean prediction. Queries with the highest variance are selected, as high variance indicates that the model's predictions are sensitive to which neurons are dropped, implying that the model has not yet learned a stable representation for these queries.

Algorithm~\ref{alg:active_learning_full} presents the complete active learning procedure.

\begin{algorithm}[htbp]
\caption{Active Learning for Cardinality Estimation}
\label{alg:active_learning_full}
\begin{algorithmic}[1]
\State Set aside 10\% of all queries as a fixed validation set
\State From the remaining queries, randomly pick 20\% as the initial training set
\State Obtain true cardinalities for the initial training set by running them on the database
\State Create a new MSCN model
\For{each active learning round}
    \State Train the model on all currently labeled queries
    \State Measure the model's accuracy on the validation set
    \State Use the chosen strategy to score each unlabeled query by how informative it would be
    \State Select the top-scoring queries and obtain their true cardinalities from the database
    \State Add the newly labeled queries to the training set
\EndFor
\State \Return the trained model and accuracy metrics
\end{algorithmic}
\end{algorithm}

The total number of queries labeled across the active learning process is $|\mathcal{L}_{\text{total}}| = \lfloor 0.2 \times |\mathcal{P}| \rfloor + R \times k_{\text{acquire}} + |\mathcal{V}|$, where $|\mathcal{P}|$ is the pool size, $R$ is the number of rounds, $k_{\text{acquire}}$ is the per-round acquisition size, and $|\mathcal{V}|$ is the validation set size. The initial labeled set comprises 20\% of the pool, providing sufficient data for the model to learn a reasonable baseline before active acquisition begins.


\section{Summary}

This chapter presented the complete training approach for the learned cardinality estimator, covering the database labeling process, the set-based query encoding with bitmap features, the MSCN neural network architecture with masked mean pooling, the Q-error loss function, and the active learning framework with three acquisition strategies. The active learning strategies aim to achieve significant improvements in sample efficiency compared to random sampling, reducing the number of expensive database executions needed to train an accurate model. The following chapter presents the experimental evaluation of the complete pipeline.
